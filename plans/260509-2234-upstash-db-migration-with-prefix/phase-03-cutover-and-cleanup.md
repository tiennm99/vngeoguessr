# Phase 3: Cutover and Cleanup

## Overview

Deploy the refactored app pointed at the NEW Upstash DB, validate end-to-end, then remove the OLD DB integration and stale dependencies. After this phase, the project depends only on the new shared Upstash DB and the `redis` npm package is gone.

## Requirements

- Production reads/writes hit the NEW DB (with prefix) — verified via real user-flow.
- OLD DB integration disconnected from the Vercel project (user-driven).
- `redis` npm dep removed from `package.json` and lockfile.
- Documentation in `docs/tech-stack.md` reflects the new SDK and prefix model.

## Architecture

No code architecture changes here — this phase is operations + documentation. Sequence:

```
1. Deploy preview with refactor (Phase 1) → smoke test against NEW DB
2. Promote preview → production
3. Validate production:
     POST /api/guess (full game flow)  → leaderboard updates in NEW DB
     GET  /api/leaderboard             → returns migrated entries
4. Disconnect OLD Upstash integration in Vercel dashboard (user)
5. Remove `redis` dep, run `npm install`, commit lockfile
6. Update docs/tech-stack.md
```

## Related Code Files

- Modify: `package.json` — remove `"redis"` dependency. **Highlight to user** per File Modification Policy.
- Modify: `package-lock.json` — regenerated by `npm install` (user runs).
- Modify: `docs/tech-stack.md` — replace "Redis: Complete session and leaderboard storage" + node-redis line with `@upstash/redis` REST SDK + `KEY_PREFIX` multi-tenant note.
- No source-code changes in this phase.

## Implementation Steps

1. **Deploy preview** (user runs `vercel` or pushes to a preview branch). Confirm preview reads from NEW DB.
2. **Smoke test on preview:**
   - GET `/api/leaderboard?type=score` → should list migrated users.
   - Play one full game → POST `/api/guess` → leaderboard rank updates.
   - Optional: check Upstash dashboard for new key like `vngeoguessr:leaderboard:vietnam` with the new score reflected.
3. **Promote to production** (`vercel --prod` or merge to `main`).
4. **Production smoke test** (same as step 2, on the production URL).
5. **Disconnect OLD Upstash integration** — user action in Vercel dashboard:
   - Project → Storage → remove the OLD Upstash integration.
   - Remove `REDIS_URL` env var if it lingers.
6. **Remove `redis` dep:**
   - User runs: `npm uninstall redis`
   - Verify `npm ls redis` is empty and `package-lock.json` no longer references `node-redis`.
7. **Update `docs/tech-stack.md`:** replace lines about node-redis with `@upstash/redis` REST SDK; add a sentence noting the `KEY_PREFIX` multi-tenant convention.
8. **Final grep sweep:**
   - `grep -r "REDIS_URL\|from 'redis'\|node:redis" src/ scripts/` → empty
   - `grep -r "vngeoguessr:" src/` → only in `src/lib/upstash.js` (the default constant)

## Success Criteria

- [ ] Production GET `/api/leaderboard` returns migrated entries from NEW DB
- [ ] Production POST `/api/guess` increments score in NEW DB (verified in Upstash dashboard)
- [ ] OLD Upstash integration removed from Vercel project (Storage tab clean)
- [ ] `redis` dep removed from `package.json`; `npm ls redis` empty
- [ ] `docs/tech-stack.md` updated to reflect `@upstash/redis` + `KEY_PREFIX`
- [ ] Final grep sweep clean (no `REDIS_URL` / `from 'redis'` references)
- [ ] `npm run build` passes on a fresh checkout

## Risk Assessment

- **Risk:** Production deploy reads from NEW DB before migration script ran ⇒ empty leaderboard for users. **Mitigation:** Phase 2 must complete and be verified before promoting Phase 3 to production. The plan ordering enforces this.
- **Risk:** OLD DB integration removed before production cutover settles ⇒ rollback impossible. **Mitigation:** keep OLD DB connected for at least one full traffic cycle (e.g., 24h) post-cutover before disconnecting.
- **Risk:** `KEY_PREFIX` env var missing in Vercel production ⇒ runtime falls back to default `vngeoguessr:`. This is fine (matches migration default), but if a different prefix was set at runtime, leaderboards appear empty. **Mitigation:** explicitly set `KEY_PREFIX=vngeoguessr:` in Vercel env vars during cutover, even though it's the default.
- **Risk:** `docs/tech-stack.md` drift (forgotten update). **Mitigation:** included in success criteria checklist; reviewer will catch.

## Out of Scope

- Adding more projects to the shared Upstash DB (the prefix system enables it; actual onboarding of other projects is separate work).
- Adopting `@upstash/redis` for any feature beyond leaderboard + session.
- Refactoring API route files — they remain untouched throughout.

## Rollback Plan

If production smoke fails after promotion:
1. In Vercel dashboard, redeploy the previous commit (one click).
2. Old commit still uses node-redis + `REDIS_URL`; OLD DB still connected → instant rollback.
3. Investigate failure offline, redo Phase 1 / 2 steps as needed before retrying.

This is why steps 5–7 (cleanup) only happen AFTER production has been stable for a full traffic cycle.
